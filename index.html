    Civic Engagement Map

    Please contact me to access this private repository.

    Final Product

    Swarthmore College has an enduring commitment to civic engagement. Civic engagement takes many forms, whether it classes that have a community impact component, grants for student-led social entrepreneurship projects, or politically-engaged research. The diversity and range of these projects is impressive, but it can be difficult to know everything your colleagues are involved in. This was the impetus to create the Civic Engagement Map. The intention was to design a tool that would allow the user to identify colleagues' projects and topic areas on campus and off. The hope was that this will foster connections between actors who identify similarities in their projects, as well as create a centralized database. By fostering connections, we hoped to encourage collaboration and more efficient resource allocation, and methods sharing. The current map prototype allows anyone with a Swarthmore ID to create a profile, log information about their project, and place pins on the map that relate to their project. The map itself is easily filtered by common topics and areas of interest.

    I began to be involved in this project in fall, 2019. I worked with Anthony Weed, a programmer in the Swarthmore IT department to continue to develop the main features of the site. My role was mostly front-end development, but supported by a strong grasp on the back-end details. Some of my contributions are as follows:

  • Keyword-search of project cards connected to filtering map pins
  • Enable admin ability to approve projects added by users
  • Highlight regions of interest and high activity on the map
  • Develop user controls : ability to create and delete a project and connect these actions to database
  • Implement user-friendly page-layout to display detials for a single project
    •  My envolvement was not solely technical. I also worked with Nhimesh Ghimire to discuss how broad project goals could be implemented in the design of the application.

    This project allowed me to gain hands-on experience with pgit strofessional tools and to help advance a project that has a palpable impact on the Swarthmore community. If I could do it again, I would have directly consulted students and faculty who intend to use the tool to have a first-hand understanding of their needs and ideas. I have found that it is quite easy to fall into a coding bubble and rely on superiors to interface with the target user community, but I think it would have been valuable to have been in direct communication with peers who plan to use the tool to gain their perspectives not only on broad objectives, but the way these objectives are translated into features.

    Bach Chorale Workshop

    Source Code

    Final Product

    I developed this project as a part of the SPEED program at Swarthmore. Music professor Andrew Hauze had identified that his students studying conducting had a hard time finding adequate tools to be develope their listening skills. For advanced musicians, it is essential to be able to pick out the different melodic and instrumental strains in a piece of music. His students faced a dilemma in developing this skill ; listening to live performances is ideal for hearing different instrumentations, but the experience is not repeatable. Listening to recording is repeatable, but difficult to pick out different strains in the audio. Thus, I worked with him to design an implement a tool that takes a broad database of Bach chorales and allows that user to control the following :
  • Instrumentation
  • Volume
  • Pausing and repeating
  • Visualizing the musical score
  • Entering into a more difficult practice mode that groupd alto and tenor parts in terms of control
    •

    I had the priviledge of writing all of the code for this project, working under the guidance of Ashley Turner and Roberto Vargas. Thus, there were multiple stages in tackling project. Firstly, I had to find or create a database of Bach chorales (which are in the creative commons) and convert the scores into MIDI format if they weren't already. Using the open-source JS library WebAudioFonts, I was able to parse each chorale into Soprano, Alto, Tenor, and Bass parts, and attach controls to each part individually. The second challenge was in procuring the musical scores, matching them to the audio files, and syncing the controls (so that if the audio tempo changed, for instance, then the score highlight should reflect the new tempo). I was able to accomplish this using another open-source library that renders the scores as an svg.

    I had to be flexible with my short-term goals in this project. I had initially thought I would have to implement each aspect from scratch, which poses it's own challenges. After finding appropriate libraries, however, the challenge became to coordinate their functionalities, and to suite them to my specific needs.

    Lastly, after establishing a viable, funcitoning tool, I worked on making it accessible. This entailed making each button, link, and drop-down menu all easily used with a screen-reader, as well as adjusting colors to make sure contrasts are high enough, and making sure that the functionality is clear to the user.
    This project was my first experience taking on the majority of the responsibility, and I learned valuable soft skills such as time management, long and short-term goal setting, and frequent communication with other project participants. I also gained a strong understanding of the website developement through learning-by-doing. If I could do it again, I would have taken accessibility into consideration as I created the elements of the app instead of wrestling with it afterwords.

    • 1847 African-American Census (Prototype)

    Source Code

    You can view the prototype here. Please contact me to see the final product, as it requires a Swarthmore log-in.

    The special collections visualization project was proposed by Celia Caust-Ellenbogen and Jordan Landes of the Friends Historical Library at Swarthmore College. The library preserved the original manuscript of a census of the 1847 African-American population in Philadelphia, collected by a committee of Quakers. We wanted to revitalize the original online exhibit for the Philadelphia 1847 Census data, in order to present the data in a streamlined, accessible setting for researchers and students. Additionlly, we wanted to add interactivity to the data and present new interpretations of the data through visualizations. This included a geocoding component to relate the 1847 addresses to contemporary Philadelphia. This would allow for a better understanding of demographic changes over time, as well as envision the history of Philadelphia's Black cummunity.

    My role in this project was to create data visualizations using the census data. I helped clean and refine the data using OpenRefine. I designed the site layout, and worked on the front-end integration of my teammates' visualizations into the site. For instance, my teammate had more expertise working with MapBox's API from another project she worked on, so she created a layered map that included a historically geocoded layer. My role was then to implement this map in the browser and create user controls to interact with the map. I also worked with another teammate to display the digitized census data and make it searchable on the site. I also created visualizations that are public on observablehq that were not included in the final site after deciding that they didn't tell a strong enough story about the data.
    I was inspired by the work of W.E.B Du Bois's visualizations at the 1900 Paris Expo. What I particularly admired about these visualizations was that they were bold and made use of familiar shapes and colors in unfamiliar patterns. I found that this both draws in the viewer, but also makes them work to discover the story being told about the data.

    There were may aspects of the data that I felt could have been interpreted or re-imagined, but that we overlooked in the interest of time. For instance, the notes section certain census entries were often blank, but sometimes were used by census-takers to describe particularly shocking living conditions or moral outrage. I thought that it would have been interesting to somehow bring the census-takers into question through our visualizations to disrupt the notion that the census is somehow objective and doesn't carry assumptions or biases. In the end, we didn't have enough time to create something to reflect this intuition. In general, this project made me think a lot about the value of representing data in familiar and unfamiliar ways. I think there is a place for both -- familiar bar-graphs and charts can intuitively reveal trends and make the data feel familiar to the viewer. Unfamiliar visualizations can draw attention to out-of-the-ordinary aspects of the data. I wish we had had time to develop more of the latter.

    Gerrymandering REU

    This project is ongoing, please check in later for deliverables!

    After the 2020 census results are released, all states in the US will be redrawing district maps based on new population data. Our goal in with this research is to first make shape files of voting precincts more transparent and accessible. Many states in the US do not currently make precinct shape files publicly available, which are often the building blocks used by state legislatures to draw district maps. By making data public and accessible in more formats, we will help democratize the redistricting process.

    We also aim to use voting and geographic data to create a basis for contesting a given district map as gerrymandered. Many measurements of gerrymandering are flawed. Take for example the Pennsylvania Seventh District, aka "Goofy kicking Donald Duck".



    This disctrict seems gerrymandered because it is a very nonstandard, noncompact shape, which indicates to us that someone went out of their way to group certain areas together that are the opposition party, a technique called packing. Thus one technique of measuring gerrymandering, the Polsby-Popper test measures the compactness of a shape. Arizona used this test in its 2000 redistricting commission with the idea that low compactness makes a stronger case for gerrymandering. This test gels well with the "you-know-it-when-you-see-it" principle, but it can be misleading and hide nuance. For example, a complication arises when respecting natural features. Additionally, many of the most offensive district shapes came from the time of hand-drawing, but computers have made it possible to ensure similar election results while maintaining 'reasonable' looking shapes. Even if you accept that the Polsby-Popper test detects gerrymandering, there is the issue of the Voting Rights Act. This act and subsequent supreme court cases establish that district maps can't disadvantage voters because of their race. Thus, there could be ambiguity about whether a district is built to 'pack' opposition voters, or whether it is designed to create an opportunity district for a minority racial population.

    The Polsby-Popper test is but one example of measurements designed to define gerrymandering that face serious flaws. What I find to be the common demoninator amongst these tests is that they try to tell the whole story by looking at one aspect of the data, and they often are based on either geographic or election data, but not both. I am very excited to be working on this research, because I believe it avoids this trap and still provides a solid basis for claiming a map to gerrymandered using outlier analysis. We are using a method developed by the Metric Geometry and Gerrymandering Group. Using a Markov Chain Monte Carlo modeling, this method creates a large random sampling of district maps. You can find a more detailed mathematical explanation here. This by simulating hypothetical elections based on recent election data on all the maps in this large sample, we can create a histogram to show the liklihood of different election results. We can then compare the map drawn by the legislatures and see where they fall on this histogram. If the election results of the legislature's map are an outlier, that builds a strong case that their map was designed to specifically advantage one part.

    The project is ongoing, but so far, I have collected and refined election and demographic data for the state of Maine and helped to merge that with precinct shape files. I have undergone training on Metric Geometry and Gerrymandering Group’s (MGGG’s) software GerryChain, so that in the next week, I can generate millions of hypothetical maps for a given state and then run hypothetical elections in them based on recent election data. This will compile a basis for creating narratives about district maps being ‘fair’ or ‘unfair’.

    Machine Learning (Spring 2020)

    Source Code

    Our objective was to write our own code in python that performs nested cross-validation and the k-nearest neighbour algorithm, estimates distances between data samples, builds confusion matrices and calculates performance metrics. I've incuded a sample of my solution to tie-breaking in k-NN, as well as an explanation:

    The way I implemented a tie-breaker in my kNN allgorithm was by weighting certain labels over others based on how near or far a neighbor they are from the unabeled sample. It also occured to me that by adding or subtracting 1 from k would easily break the tie, but this method is more likely to have issues if there are more than 2 classes (as in this case) and doesn't accurately represent the best-fit k parameter used to label the data, which is important in cross-validation. The way I implemented getNeighbors() meant that I already had a list of k neighbors sorted by distance with the nearest neighbor at index 0. So for a range of i = 0 ... k, I added the weight k-i to the label so that the the nearest neighbor would have the highest weight of k and the farthest neighbor would have the lowest weight of k - (k-1). I then computed the cumulative weight for each label and returned the 'heaviest'. The code I used in mykNN() to accomplish this I have copied into the cell below. To further optimize the accuracy of tie-breaking, I could implement multiple tie-breaking algorithms and include a parameter in myNestedCrossVal() to optimize the tie-breaking method used similar to how k and the distance function are optimized. 
                                                                  
def breakTie(nLabels):

label_weights = []
for i in range(0, len(nLabels)):
    w = len(nLabels) - i
    lw = (nLabels[i], w)
    label_weights.append(lw)
label_weights.sort(key= lambda x:x[0])

weighted_frequencies = {}
curr_tar = -1
for lw in label_weights:
    if lw[0] == curr_tar:
        weighted_frequencies[lw[0]] += lw[1]
    else:
        curr_tar = lw[0]
        weighted_frequencies.update({lw[0]: lw[1]})
mode = max(weighted_frequencies, key=weighted_frequencies.get)
return mode

    Our objective was to implement logistic regression in python using batch gradient descent. I have copied my write-up below:

    The main entry point to my solution is the gradient_descent() function. This function uses the batch gradient descent technique to calculate optimized theta values for a logistic hypothesis function. The function is an appropriate model for a binary classification problem. Given theta values, predict() will assign target values to each data sample in X based on a logistic hypothesis function parameterized on theta that is determined by calling h_log().

    The function gradient_descent() relies on functions gradient_batch() and loss(). The loss() function in turn relies on the cost() function because to find the cost for one data sample in X. The loss() function then returns the average cost by summing all costs and dividing by the number of samples. The cost() function calculates cost using cross entropy, which ensures convergence to a global minimum. It relies on h_log() to calculate the cost for a given data sample. The gradient_descent() function calculates new theta values based on old theta values, a learning rate, and a gradient determined in gradient_batch() until the loss (calculated by loss() ) of using the new theta values in a hypothesis function is less than a given threshold.

    The gradient_batch() function depends on h_log() to calculate the gradient for a given theta_j. The gradient is calculated by summing for all samples the differences between h_log() probability and real target values multiplied by the sample's predictor value for feature j.

    The hypothesis function h_log() only depends on inner(). The logistic hypothesis function h_theta(x) can be thought of as g(theta.transpose * x), where g(x) is the sigmoid function. Thus in my implementation, g(x) corresponds to the function h_log(), and the input to g(x) is calculated in the function inner(). Note: because of the data structures I use in my implementation, inner() does not explicitly use the transpose of theta, but the return value is equivalent to the dot product of theta transpose and x.
